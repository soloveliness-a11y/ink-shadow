# 增量状态同步（JSON Patch）

## 目标

讨论期每次玩家行动（发言/搜证/公开线索）触发的 stateSync 从全量快照（25-60KB）改为增量 patch（1-3KB），减少 70%+ 带宽，降低客户端频繁全量替换 state 的 GC 压力。

## 现状分析

- `broadcastState()` 每次为每个在线玩家调用 `buildView()` 生成完整 `ClientStateView`（24 个顶层字段）
- Free 讨论期最热路径：每秒 3-6 次全员广播，每次 25-60KB
- 无压缩、无增量、无缓存
- 高频变化字段仅 6-8 个（`log`、`phaseProgress`、`currentPhase.mySearchCount`、`searchableClues`、`sceneSearchProgress`、`self.myClues`、`revealedClues`、`counters`）
- 静态字段 16+ 个（`publicCharacters`、`publicScenes`、`selectedScript`、`self.privateScript` 等）

## 方案：服务端 diff + statePatch 协议

### 核心思路

1. 服务端为每个连接的客户端缓存上一次发送的 view 快照
2. 广播时先 buildView 生成新 view，再与缓存的旧 view 做 JSON diff
3. diff 为空则跳过（无变化），否则发送 `statePatch` 消息（只含变更路径）
4. 客户端收到 `statePatch` 后 merge 到本地 view
5. 关键节点（重连、phase 切换、首次 join）仍发全量 `stateSync` 作为锚点

### diff 格式（自定义，轻于 RFC 6902）

```ts
interface StatePatch {
  kind: 'statePatch';
  /** 变更路径 → 新值。路径用 JSON Pointer（如 '/log'、'/phaseProgress/actedCount'） */
  patches: Record<string, unknown>;
  /** 删除的路径（值为 null/undefined 的叶子节点） */
  removes?: string[];
}
```

只传变更的叶子节点路径，不做数组 diff（`log` 追加场景直接传整个新 `log` 数组）。

## 实现步骤

### Step 1: Schema 协议扩展

**文件**: `packages/schema/src/protocol.ts`

新增 `statePatch` 消息类型：

```ts
// zServerMessage discriminated union 新增分支
z.object({
  kind: z.literal('statePatch'),
  patches: z.record(z.string(), z.unknown()),
  removes: z.array(z.string()).optional(),
})
```

同步更新 `ServerMessage` 类型。

### Step 2: 服务端 diff 引擎

**新建文件**: `packages/server/src/diff.ts`

```ts
/** 比较两个 view，返回变更路径映射。空对象表示无变化。 */
export function diffViews(prev: Record<string, unknown>, next: Record<string, unknown>): {
  patches: Record<string, unknown>;
  removes: string[];
}
```

实现策略：
- 只做顶层 + 二级字段 diff（不递归到叶子节点，避免 CPU 开销）
- 对于数组字段（`log`、`players`、`publicCharacters`）：引用相同则跳过，不同则全量替换
- 对于对象字段（`self`、`currentPhase`、`phaseProgress`）：逐子字段比较
- 对于原始字段（`roomCode`、`status`）：直接比较

### Step 3: Room 广播改造

**文件**: `packages/server/src/room/Room.ts`

```ts
// 每个 playerSocket 增加 cachedView 字段
private playerSockets = new Map<string, { charId?: string; cachedView?: Record<string, unknown> }>();

// broadcastState → fanOutStateSync 改造
private fanOutStateSync(shared: ReturnType<typeof buildSharedParts>): void {
  for (const [playerId, sock] of this.playerSockets) {
    if (!sock.charId) continue; // 未分配角色跳过
    const p = this.state.players.find(p => p.playerId === playerId);
    if (!p?.connected) continue;
    const newView = buildView(this.script, this.state, playerId, this.availableScripts, this.viewExtra(), shared);
    const newViewObj = newView as unknown as Record<string, unknown>;
    const prevView = sock.cachedView;
    if (prevView) {
      const { patches, removes } = diffViews(prevView, newViewObj);
      if (Object.keys(patches).length > 0 || removes.length > 0) {
        this.sendToPlayer(playerId, { kind: 'statePatch', patches, removes });
      }
    } else {
      // 首次或缓存丢失 → 全量
      this.sendToPlayer(playerId, { kind: 'stateSync', view: newView });
    }
    sock.cachedView = newViewObj;
  }
}
```

**强制全量的场景**（清空 cachedView）：
- `join()` 成功后
- `handleDisconnect()` 重连后
- PhaseEngine `enter()` 进入新 phase 时（phase 切换时大量字段变化，全量更高效）
- `rollbackSnapshot()` 回退快照后

### Step 4: 客户端 patch 合并

**文件**: `packages/client/src/store/game.ts`

```ts
case 'statePatch': {
  setState((s) => {
    if (!s.view) return {}; // 无 base view 时忽略 patch
    const view = structuredClone(s.view) as Record<string, unknown>;
    // 应用 patches
    for (const [path, value] of Object.entries(msg.patches)) {
      setByPath(view, path, value);
    }
    // 应用 removes
    for (const path of msg.removes ?? []) {
      deleteByPath(view, path);
    }
    return { view: view as ClientStateView };
  });
  break;
}
```

辅助函数 `setByPath` / `deleteByPath`：按 JSON Pointer 路径设置/删除值。

### Step 5: 性能优化（可选后置）

- **WS permessage-deflate**：`new WebSocketServer({ perMessageDeflate: true })` 已在 index.ts 中启用，但当前可能未生效，需确认
- **log 增量追加**：对于 `log` 数组，如果只追加了新元素，可以只发 `{ '/log/-': newItem }` 而非全量替换
- **节流**：高频场景下（>5次/秒）合并最近的 patch 为一次发送
